hc——通过拓展残差流的宽度和丰富连接模式，但是破坏了残差恒等映射属性（残差恒等映射属性指的是较浅层的信号会直接映射到较深层，且不做任何修改），从而导致了当训练step较大时，训练的不稳定性和拓展性受限，也导致了内存访问开销的增大。
mhc——将hc残差连接空间投影到特定流形上以恢复恒等映射属性。



 一、背景知识：MLA 和矩阵吸收（MLA Absorb）

  DeepSeek 用了一种叫 MLA（Multi-head Latent Attention） 的注意力机制。标准 MHA 每个 head 都有独立的 K、V 矩阵，而 MLA
   用了低秩压缩：

                   ┌──────────────────────────────────────────┐
                   │  输入 x [b, s, dim=7168]                  │
                   │                                          │
                   │  wkv_a: [dim → kv_lora_rank+rope_dim]    │
                   │  压缩成低维潜变量 kv [b,s,kv_lora_rank=512] │
                   │                                          │
                   │  wkv_b: [kv_lora_rank → n_heads*(nope+v)]│
                   │  再展开成每个 head 的 K 和 V                │
                   └──────────────────────────────────────────┘

  矩阵吸收（MLA Absorb） 是一个推理优化技巧：
  - 普通做法：先把 kv 用 wkv_b 展开成完整的 V [b,s,n_heads,v_head_dim]，再做注意力
  - 吸收做法：注意力计算完，得到一个中间结果 output，延迟投影 V，即在注意力之后再乘上 w_uv_t

  这样 kv cache 只需存 512 维的压缩向量，而不是 128 个 head × 128 维的完整 V。

  ---
  二、w_uv_t 是什么

  # PreAttention.forward（MLA absorb 路径）
  wkv_b_weight = self.wkv_b.weight.reshape(
      -1,                           # 这一维 = n_heads = 128
      self.qk_nope_head_dim + self.v_head_dim,  # 128 + 128 = 256
      self.kv_lora_rank             # 512
  )
  # 拆分 nope 部分和 v 部分
  w_uk = wkv_b_weight[:, :self.qk_nope_head_dim, :]  # [128, 128, 512]  用于 K
  w_uv = wkv_b_weight[:, self.qk_nope_head_dim:, :]  # [128, 128, 512]  用于 V

  w_uv_t = w_uv.permute(0, 2, 1).contiguous()        # [128, 512, 128]  转置后传出

  所以 w_uv_t 的形状是 [n_heads, kv_lora_rank, v_head_dim] = [128, 512, 128]，它是 wkv_b
  权重中负责"V投影"的那一半，转置后备用。

  w_uv_t 来自 self.wkv_b.weight，而 wkv_b 被配置为
  colwise_parallel(use_local_output=False)（parallelize.py:466），意味着这个权重矩阵是一个 DTensor，在 TP 维度上分片。

  ---
  三、output 是什么

  inner_attention (DSASparseAttention / DSV32_SDPA):
    输入:  q [b, n_heads, s, qk_head_dim]
           k [b, 1, s, kv_lora_rank]     ← 压缩的 k（MLA absorb 模式下）
           v [b, 1, s, kv_lora_rank]     ← 压缩的 v（同上）
    输出:  output [b, n_heads, s, kv_lora_rank]
           ↑ 这个不是最终的 V，而是注意力权重加权后的"压缩空间"表示

  在 MLA absorb 模式下，注意力核做的事情是：

  output[b,h,s,r] = sum_over_s'(  softmax(q[b,h,s,:] · k[b,0,s',:]) · v[b,0,s',r]  )

  得到的 output 形状是 [b, n_heads, s, kv_lora_rank]（而不是 [b, n_heads, s, v_head_dim]），还需要再乘 w_uv_t
  才能得到最终 V 空间的结果。

  ---
  四、einsum 在做什么

  output = torch.einsum("bhsr,hrv->bhsv", output, w_uv_t)

  逐字母解读：
  - b = batch size
  - h = head 数量（128）
  - s = sequence length
  - r = kv_lora_rank（512，压缩维度）
  - v = v_head_dim（128）

  操作：对每个 (b, h, s) 位置，做一个向量乘矩阵：

  output_new[b,h,s,v] = sum_r( output[b,h,s,r] × w_uv_t[h,r,v] )
                              ↑ 注意力得到的压缩结果    ↑ 第 h 个 head 的 V 投影矩阵

  关键点：h 维度在两边都出现但不求和（没有出现在 → 右侧的前提下它是保留维度），这意味着每个 head
  用自己专属的那块权重矩阵，各 head 完全独立计算，不跨 head 交互。

  ---
  五、什么是 DTensor，什么是 to_local()

  DTensor（Distributed Tensor） 是 PyTorch 分布式并行的核心抽象。它把一个逻辑上的大 tensor 分散存放在多个 GPU
  上。每张卡只保存一个分片（shard），但 DTensor 对外表现出完整的形状。

  全局 tensor w_uv_t 形状: [128, 512, 128]   ← DTensor 的"逻辑形状"
                           ─────────────────────────────
                           Rank 0 本地存储:  [64, 512, 128]   ← Shard(0) 分片
                           Rank 1 本地存储:  [64, 512, 128]   ← 另一半

  to_local() 的作用：把 DTensor 变成该 rank 上实际存储的普通 tensor（不再有分布式语义）：

  w_uv_t_dtensor  # 形状显示为 [128, 512, 128]（逻辑形状）
  w_uv_t_local = w_uv_t_dtensor.to_local()  # 形状变为 [64, 512, 128]（真实本地数据）

  ---
  六、TP（Tensor Parallel）如何切分 Attention

  使用 TP=2 时，注意力模块的权重按 head 维度分片：

                    TP Rank 0                    TP Rank 1
                   ┌──────────────┐             ┌──────────────┐
    wq_b           │ heads 0~63   │             │ heads 64~127 │   ColwiseParallel
    wkv_b          │ heads 0~63   │             │ heads 64~127 │   ColwiseParallel
    (因此 w_uv_t)  │ [64, 512,128]│             │ [64, 512,128]│   DTensor Shard(0)
                   └──────────────┘             └──────────────┘

    q              │ [b,64,s,192] │             │ [b,64,s,192] │   各自算各自的 head
    output(attn)   │ [b,64,s,512] │             │ [b,64,s,512] │   各自的注意力结果

  每个 rank 只负责 n_heads/TP = 64 个 head，这样两个 rank 并行，相当于同时算完所有 128 个 head。

  ---
  七、bug 是如何触发的

  现在来看关键的数据流：

  Step 1：inner_attention 的输出包装

  parallelize.py:398 配置了 attention_kernel_plan：
  attention_kernel_plan = PrepareModuleInputOutputWithBwdAllReduce(
      use_local_input=True,          # 输入用本地 tensor（用本地的 q/k/v）
      output_layouts=(Replicate(), Shard(1)),
      desired_output_layouts=(Replicate(), Shard(1)),
      use_local_output=False,        # ← 输出包装成 DTensor，不取本地
  )

  use_local_output=False 的意思：注意力核返回的普通 tensor，会被这个"计划"重新包装成 DTensor，告诉系统"这个 tensor 在
  dim=1（head维）上是分片的"。

  结果：output 从注意力核出来时是 [b, 64, s, 512] 的普通 tensor，但被包装成一个 DTensor，逻辑形状 [b, 128, s,
  512]，Shard(1)。

  Step 2：w_uv_t 的来源

  wkv_b 是 colwise_parallel(use_local_output=False)，它的 weight 是 DTensor。从这个权重重塑、切片、转置得到
  w_uv_t，也是一个 DTensor，逻辑形状 [128, 512, 128]，Shard(0)。

  Step 3：错误的 to_local() 组合

  if isinstance(w_uv_t, DTensor):
      w_uv_t = w_uv_t.to_local()   # ✓ 变成普通 tensor [64, 512, 128]
  # output 依然是 DTensor！逻辑形状 [b, 128, s, 512]
  output = torch.einsum("bhsr,hrv->bhsv", output, w_uv_t)
  #                                        ↑ DTensor   ↑ 普通 tensor

  当 PyTorch 遇到 DTensor + 普通 tensor 做运算时，它的处理策略是：先把 DTensor All-Gather（跨卡收集）成完整的
  Replicate 状态，再和普通 tensor 计算。

  All-Gather 之后：
  - output：逻辑形状 [b, 128, s, 512] → 在每张卡上都变成完整的 [b, 128, s, 512]
  - w_uv_t：已经是普通 tensor [64, 512, 128]（to_local 之后就不管 TP 了）

  einsum("bhsr,hrv->bhsv")
         ↑ h=128          ↑ h=64
              ← 尺寸冲突！→
  RuntimeError: subscript h has size 64 for operand 1 which does not broadcast with previously seen size 128

  ---
  八、为什么加上 output.to_local() 就修复了

  两个 DTensor 都取本地分片后：

  Rank 0:                                 Rank 1:
  output.to_local()  → [b, 64, s, 512]   [b, 64, s, 512]
  w_uv_t.to_local()  → [64, 512, 128]    [64, 512, 128]

  einsum("bhsr,hrv->bhsv")
         h=64         h=64   ✓ 匹配！

  结果:  [b, 64, s, 128]                  [b, 64, s, 128]

  这在数学上是完全正确的：因为 h 维度在 einsum 中是逐元素对应（不跨 head 求和），每个 head
  只用自己的权重块独立计算，所以每张卡单独算自己负责的 64 个 head，结果和整体计算一模一样。

  ---
  九、为什么其他地方不需要 to_local()

  这个问题问得很关键。我们来看其他涉及 DTensor 的位置：

                      TP Rank 0 的数据流
  ─────────────────────────────────────────────────────────────────
  wkv_a (NoParallel)        → 普通线性层，输入/输出不分片
  ↓
  kv (DTensor, Replicate)   → 每张卡都有完整的压缩 kv
  ↓
  wkv_b (ColwiseParallel)   → 按 head 切分输出，但在 MLA absorb
                              路径下，wkv_b 只用来提取权重，
                              不用来做 kv 的前向推理
  ↓
  w_uv_t (DTensor, Shard)   → 需要 to_local() ← 我们处理的
  ↓
  inner_attention           → 输入是本地 q/k/v（use_local_input=True）
                              输出被包成 DTensor Shard(1) ← 需要 to_local() ← 我们处理的
  ↓
  einsum（修复后）           → 两个本地 tensor，直接算
  ↓
  wo (RowwiseParallel)      → 这个层设计来接受"分片输入"，
                              内部自动做 AllReduce，不需要手动处理
  ─────────────────────────────────────────────────────────────────

  wo 不需要处理的原因： wo 被配置为 rowwise_parallel(output_layouts=Shard(1))，这是 PyTorch TP
  的标准层。RowwiseParallel 的设计就是接收 last-dim 分片的本地输入，内部自动做 AllReduce 汇总所有 rank
  的贡献，产生正确的结果。这是 TP 的常规用法，框架已经处理好了。

  只有 einsum 需要手动处理的原因： einsum 是自定义算子，不在 DTensor 的标准并行策略里。当它遇到 DTensor × 普通 tensor
  时，不知道"我应该用本地分片算就行"，只会保守地 AllGather 成完整 tensor
  再算，导致维度不匹配。所以这里需要在调用之前手动把两个操作数都变成本地 tensor，告诉 PyTorch "别管分布式了，直接算"。
