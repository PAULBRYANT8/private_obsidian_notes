## 3. 问题演进过程

### 3.1 错误 1: Activation Checkpointing 副作用

#### 3.1.1 错误现象

torch._dynamo.exc.Unsupported: HOP: Unsafe side effect

  Higher Order Operator: torch.utils.checkpoint.checkpoint

  Explanation: Mutating a variable from outside the scope of this HOP is not supported.

Traceback:

  File "torchtitan/distributed/expert_parallel.py", line 134, in _token_dispatch

    self.input_splits = input_splits.tolist()

  File "torchtitan/distributed/expert_parallel.py", line 135, in _token_dispatch

    self.output_splits = output_splits.tolist()

#### 3.1.2 错误原因分析

**问题代码**（`expert_parallel.py:134-135`）：

class ExpertParallel:

    def __init__(self):

        self.input_splits = None  # 实例变量

        self.output_splits = None

    def _token_dispatch(self, mod, inputs, device_mesh):

        # 在 checkpoint 内部修改实例变量

        self.input_splits = input_splits.tolist()   # ❌ 副作用！

        self.output_splits = output_splits.tolist() # ❌ 副作用！

**为什么是副作用？**

Activation Checkpointing (`torch.utils.checkpoint.checkpoint`) 的工作原理：

1. Forward 时不保存中间激活值
2. Backward 时重新计算 forward
3. **要求 forward 函数是纯函数**（无副作用）

`self.input_splits = ...` 修改了外部实例变量，违反了纯函数约束。

#### 3.1.3 解决方案与结果

**采用方案: 禁用 Activation Checkpointing** ✅

# 添加参数

--activation_checkpoint.mode none

**运行结果**：

[titan] 2026-02-03 15:42:55 - Training starts at step 1

✅ 第 1 次迭代成功完成（Warmup 阶段）

**原因**：禁用 AC 后，不再有纯函数约束，允许修改实例变量。

### 3.2 错误 2: CUDA Graph Pinned Memory

#### 3.2.1 错误现象

RuntimeError: Cannot copy between CPU and CUDA tensors during CUDA graph 

capture unless the CPU tensor is pinned.

发生时机: 第 2 次迭代（CUDA Graph Capture 阶段）

Traceback:

  File "cudagraph.py", line 137

    _to_copy_29 = torch.ops.aten._to_copy.default(..., device=device(type='cpu'))

#### 3.2.2 原因分析

**问题代码**：

input_splits = num_tokens_per_expert.sum(dim=1).to(torch.device("cpu"))

# ❌ GPU → CPU 传输，使用普通 CPU 内存（非 pinned）

**CUDA Graph 的要求**：

- 录制时记录内存地址
- 重放时地址必须相同
- 普通 CPU 内存地址可能改变（操作系统可交换）
- **必须使用 pinned memory**（地址固定）

#### 3.2.3 尝试方案: 使用 pin_memory()

**代码修改**：

input_splits = torch.empty(..., device="cpu").pin_memory()

input_splits.copy_(input_splits_gpu)

**运行结果**：❌ 失败

TorchRuntimeError: NYI: aten._pin_memory.default

**失败原因**：`pin_memory()` 在 TorchDynamo 的 FakeTensor 模式下未实现，无法进行图捕获。

---

### 3.3 错误 3: torch.compiler.disable 不被允许

#### 3.3.1 错误现象

torch._dynamo.exc.Unsupported: Skip calling `torch.compiler.disable()`d function

Hint: Remove the `torch.compiler.disable` call

#### 3.3.2 原因分析

**fullgraph_capture 模式的严格要求**：

# graph_utils.py:52

gm = dynamo_graph_capture_for_export(model)(*args, **kwargs)

这是 **fullgraph capture** 模式：

- ❌ 禁止任何 graph break
- ❌ 禁止 `torch.compiler.disable`
- ❌ 禁止动态控制流

**为什么这么严格？**

CUDA Graph 需要完整的、连续的 GPU 操作序列。任何 graph break 都会破坏 CUDA Graph 的连续性。

#### 3.3.3 根本性不兼容

**技术栈冲突**：

DeepSeek V3 需求                    CUDA Graph 要求

├─ 动态路由                         ├─ 静态图结构

├─ List[int] splits                ├─ 只处理 Tensor

├─ .tolist() 转换                  ├─ 禁止 Python 数据结构

└─ GPU → CPU 传输                  └─ 禁止设备间传输

         ❌ 架构级不兼容

## 4. CUDA Graph 技术原理

### 4.1 传统执行模式的开销

每个 GPU kernel 调用:

Python → C++ → CUDA API → GPU

开销分析:

- Python 调用: ~10 μs

- Kernel 启动: ~8 μs

- 实际计算: ~50 μs

500 个 kernels:

- 总开销: 500 × 18 = 9000 μs

- 占比: 约 15-30%

### 4.2 CUDA Graph 优化机制

**三阶段执行**:

第 1 次迭代（Warmup）:

- 正常执行

- 收集元数据

- 时间: 正常

第 2 次迭代（Capture）:

- 录制所有 GPU 操作

- 记录内存地址

- 时间: 较慢（+50%）

第 3+ 次迭代（Replay）:

- cuda_graph.replay()  # 单次调用

- GPU 自动执行所有 kernels

- 时间: 快 20-35%

**性能收益**:

- Python 开销: 10μs × 500 → 10μs ✅
- Kernel 启动: 8μs × 500 → ~0μs ✅
- 总提升: **20-35%**

### 4.3 CUDA Graph 的限制

**可以录制**:

- ✅ GPU kernel 调用
- ✅ GPU 内存操作
- ✅ NCCL 通信（GPU 间）

**无法录制**:

- ❌ CPU-GPU 数据传输
- ❌ Python 数据结构转换（.tolist()）
- ❌ 动态控制流
- ❌ 动态内存分配

## 5. 核心兼容性问题分析

### 5.1 为什么 Llama 可以，DeepSeek V3 不行？

#### Llama 架构（Dense）

def forward(self, x):

    # ✅ 100% GPU 操作

    x = self.attention(x)  # GPU

    x = self.mlp(x)        # GPU (单个 FFN)

    return x               # GPU

训练循环:

inputs = batch.to('cuda')  # ← 唯一 CPU → GPU (图外)

outputs = model(inputs)    # ✅ 全程 GPU

loss.backward()            # ✅ 全程 GPU

# 无 GPU → CPU 传输！

#### DeepSeek V3 架构（MoE）

def forward(self, x):

    # 路由决策

    topk_indices = self.gate(x).topk(k=3)  # GPU

    # 统计 tokens

    num_tokens = count_per_expert(topk_indices)  # GPU

    # ❌ 关键问题

    splits = num_tokens.to('cpu').tolist()

    #        ^^^^^^^^^^^^^^ GPU → CPU

    #                      ^^^^^^^^ Python list

    # all-to-all 通信

    tokens = all_to_all_single(

        tokens,

        input_split_sizes=splits,  # List[int] 要求

    )

**对比**:

|模型|CPU-GPU 传输|原因|CUDA Graph|
|---|---|---|---|
|Llama|❌ 无|Dense FFN|✅ 兼容|
|DeepSeek V3|✅ **有**|MoE + EP|❌ 不兼容|

### 5.2 为什么 `.tolist()` 是必需的？

#### NCCL API 限制

// NCCL C++ 接口

for (int i = 0; i < world_size; i++) {

    ncclSend(..., send_counts[i], ...);

    //            ^^^^^^^^^^^^^^^^ 必须是 CPU int[]

}

**关键**：

- NCCL 需要 host memory 的 split sizes
- 无法接受 GPU tensor
- 这是库设计限制

#### PyTorch API

def all_to_all_single(

    input_split_sizes: Optional[List[int]],  # ← List[int]

):

    # 内部转为 C++ vector<int>（host memory）

    cpp_splits = list_to_cpp_vector(input_split_sizes)

**无法改变**的原因：

1. 类型系统：API 声明 `List[int]`
2. C++ 绑定：期望 `std::vector<int>`
3. NCCL 协调：CPU 端完成

## 6. 并行策略深度解析

### 6.1 三种并行策略

#### Data Parallel (DP)

切分维度: Batch

GPU 0: Model(完整) + Data[0:8]

GPU 1: Model(完整) + Data[8:16]

#### Tensor Parallel (TP)

切分维度: 模型参数

GPU 0: W[:, 0:2048]

GPU 1: W[:, 2048:4096]

#### Expert Parallel (EP)

切分维度: MoE Experts

GPU 0: Expert[0-3]

GPU 1: Expert[4-7]

### 6.2 EP 和 DP 的关系

**关键**: EP 复用 DP 的 device mesh

ep_degree == dp_shard_degree  # 必须相等

NGPU = dp_degree × tp_degree  # EP 不额外占用

**示例**（您的配置）:

NGPU = 4

DP = 2, TP = 2, EP = 2

验证:

- 4 = 2 × 2 ✅

- EP (2) == DP (2) ✅

### 6.3 完整 Device Mesh

┌─────────────────────────────────┐

│ DP/EP Group 0                   │

│ GPU 0: TP=0, Data[0], Exp[0-3] │

│ GPU 1: TP=1, Data[0], Exp[0-3] │

└─────────────────────────────────┘

       ↕ DP All-Reduce

       ↕ EP All-to-All

┌─────────────────────────────────┐

│ DP/EP Group 1                   │

│ GPU 2: TP=0, Data[1], Exp[4-7] │

│ GPU 3: TP=1, Data[1], Exp[4-7] │

└─────────────────────────────────┘

  ↔ TP All-Reduce

## 7. 最终结论与建议

### 7.1 核心发现

**CUDA Graph 与 DeepSeek V3 根本性不兼容**：

1. MoE + EP 需要 GPU → CPU 传输
2. fullgraph_capture 禁止任何 graph break
3. PyTorch API 限制无法绕过

### 7.2 性能方案对比

|方案|提升|难度|推荐|
|---|---|---|---|
|CUDA Graph|20-35%|极高|❌|
|Auto-bucketing|15-25%|简单|✅|
|Regional Inductor|10-20%|简单|✅|
|**组合**|**30-40%**|简单|✅✅|

### 7.3 推荐配置

# 最佳实践

./run_train.sh \

  --model.name compiler_toolkit.deepseek_v3 \

  --parallelism.data_parallel_shard_degree=2 \

  --parallelism.tensor_parallel_degree=2 \

  --parallelism.expert_parallel_degree=2 \

  --activation_checkpoint.mode none \

  --compile.passes autobucketing_reordering,regional_inductor

  # ↑ 不用 cudagraph

**预期结果**:

- ✅ 稳定训练
- ✅ 性能提升 30-40%
- ✅ 无编译错误

### 7.4 关键经验

1. **不是所有优化都适用所有模型**
    
    - Dense → CUDA Graph 好
    - Sparse MoE → CUDA Graph 不兼容
2. **组合优化更稳定**
    
    - 多个中等优化 > 单个激进优化
3. **理解底层机制很重要**
    
    - 知道限制，避免无效尝试

### 7.5 术语表

|术语|英文|解释|
|---|---|---|
|页锁定内存|Pinned Memory|物理地址固定的 CPU 内存|
|图捕获|Graph Capture|CUDA Graph 录制过程|
|全图捕获|Fullgraph Capture|不允许 graph break 的严格模式|
